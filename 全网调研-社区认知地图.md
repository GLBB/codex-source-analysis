## 1. 稳定共识

### Codex 已从 CLI 演进为多 Surface Runtime

社区早期主要讨论终端 CLI；当前源码已经包含 TUI、exec、App Server、daemon、SDK、Cloud Tasks、remote control 和 remote environments。只用“Rust CLI”概括已不充分。

### Agent Harness 与模型同等重要

工具规划、Prompt 缓存、compaction、审批、沙箱、rollout 和 UI 共同决定实际效果。模型 benchmark 不能单独解释长任务可靠性。

### MCP 成为基础扩展协议

Codex、Claude Code、OpenCode、goose 和 Continue 均提供 MCP 路径。差异已转向：

-工具命名与 schema 清洗；
- direct/deferred exposure；
- OAuth；
- elicitation；
- Plugin 打包；
-审批与沙箱如何包住 MCP。

### 长会话需要显式状态

Resume、fork、rollback、compaction 和 memory 都要求事件或状态存储。仅把最近聊天消息保存到数组无法支撑当前产品面。

### 安全不是一个模式开关

主流产品都至少提供 permission/confirmation；差异在是否继续提供：

- OS sandbox；
-网络强制；
- Hook/policy；
-企业 managed layer；
-自动 reviewer；
-审计与恢复。

## 2. 主要争议

| 争议 | 一侧 | 另一侧 | 本研究结论 |
| --- | --- | --- | --- |
| 默认自治还是默认询问 | 降低打断 | 降低副作用风险 | 取决于仓库与凭证威胁模型 |
| 全量工具还是按需发现 | 模型立即可用 | 降低 schema/token | 工具生态变大后 deferred exposure 必要 |
| 单 Agent 还是 Multi-Agent | 简单、成本低 | 上下文隔离、并行 | 子任务需要有界结果和审批桥 |
| Git checkpoint 还是事件溯源 | 易审查文件变化 | 可恢复完整 runtime | 两者解决不同状态层 |
| MITM 网络治理 | 方法级审计 | 证书与隐私风险 | 只适合明确受管环境 |
| Provider-agnostic | 模型选择自由 | 难用深层能力 | Harness 目标决定取舍 |
| Plugin marketplace | 易分发 | 供应链风险 | 必须有安装策略、来源和大小/路径校验 |

## 3. 常见误解

### “一次用户输入就是一次模型请求”

工具、追加输入、Hook 续写和 compaction 都可能使一个 Turn 多次采样。

### “开启 approval 就安全”

Approval 解决授权；sandbox 解决实际资源边界。用户可能批准一个表面安全、内部脚本危险的命令。

### “Plugin 就是 MCP”

Plugin 是分发包，可以含 Skills、MCP、Apps 和 Hooks。MCP 是工具/资源/elicitation 协议。

### “Rollout 就是发给模型的消息”

Rollout 保存事件事实；模型 history 是重放和投影后的当前有效视图。

### “远程 Provider 等于远程执行”

模型后端、App Server、Cloud Task 和 Exec Environment 是四种不同远程边界。

### “Code Mode 绕过工具安全”

Code Mode 的 nested calls 仍进入 Tool Router、Registry、approval 与 sandbox。

## 5. 同类项目坐标

### Claude Code

官方资料显示其核心能力包括终端/IDE/桌面/浏览器、Agent SDK、Hooks、Subagents、MCP、Plugins、permissions、sandbox 和 auto memory。它与 Codex 的共同方向是把 CLI harness 产品化为多 Surface Agent。

### OpenCode

官方架构是 TUI + HTTP/OpenAPI Server，并提供 SDK 与 ACP。它强调多 Provider、Agents、Skills、Plugins、MCP 和 pattern permissions。

### goose

已迁入 Linux Foundation 的 Agentic AI Foundation，提供 Rust desktop/CLI/API。Extension 建立在 MCP 上，默认自治，并内置 Code Mode、Memory、Summon 和 task tracking。

### Aider

核心仍是显式文件上下文、repo map、编辑格式、Git commit/undo 和 Architect/Editor 两阶段。它不是“缺少通用 Agent 特性”，而是选择了更专注的 pair-programming 边界。

### Continue

官方项目已发布最终 2.0.0，仓库只读。其遗产包括 IDE Agent、CLI、MCP、Rules 和简单的 Ask First/Automatic/Excluded tool policy。

## 6. 研究方法

### 源码事实

必须能定位到当前基线中的：

-文件；
-类型/函数；
-测试；
-生成 schema；
-命令输出。

### 历史事实

用 commit/PR/issue 说明“何时变化”，不能将旧实现写成当前事实。

### 外部事实

涉及同类产品时优先官方文档，并标注复核日期。若资料只证明应用层 permission，不能推断存在 OS sandbox。

### 设计推断

使用“可能、体现、可解释为”，并说明依据。不要把源码结构反推成官方动机。
