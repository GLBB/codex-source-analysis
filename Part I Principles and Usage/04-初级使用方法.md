# 第 04 章：初级使用方法

> 源码基线：`upstream/main@283bc4cf011047314b4804c0f1ccd06e4f6a95c5`，复核日期：2026-06-24。

## 1. 初次使用的真实链路

“安装后输入问题”只描述了表面。一次可靠的首次使用包含：

```text
安装 native binary
  -> 选择登录方式并安全存储凭据
  -> 选择 TUI 或 exec
  -> 确认 cwd、Git 仓库和 project trust
  -> 选择 permission/approval/sandbox
  -> 观察 thread/turn/tool 输出
  -> 退出后决定是否 resume
```

初学者最容易出错的地方不是 prompt 写法，而是凭据、工作目录和权限边界。

## 2. 安装与版本确认

npm/bun 安装最终启动 Rust native binary。安装后先确认：

```bash
codex --version
codex --help
codex doctor
```

`codex doctor` 比只检查 `which codex` 更有价值，因为它会诊断配置、auth 和 runtime 健康状态。

如果 npm 包存在但平台可选依赖缺失，JS launcher 会建议重新安装 `@openai/codex@latest`。行为异常时应同时检查：

- shell 实际命中的 `codex` 路径。
- native binary 版本。
- `CODEX_HOME`。
- 安装是 npm、bun 还是 standalone。

## 3. 登录

当前 CLI 支持四类入口：

| 方式 | 命令形态 | 适用场景 |
| --- | --- | --- |
| ChatGPT 浏览器登录 | `codex login` | 有浏览器和 localhost 回调的个人交互环境。 |
| Device code | `codex login --device-auth` | SSH、容器或浏览器回调不便的环境。 |
| API key stdin | `printenv OPENAI_API_KEY \| codex login --with-api-key` | API 计费和自动化环境。 |
| Access token stdin | `printenv CODEX_ACCESS_TOKEN \| codex login --with-access-token` | 受控 token 注入场景。 |

旧的 `--api-key <value>` 已被拒绝，目的是避免密钥进入 shell history 和进程参数。API key/access token 应从 stdin 输入。

### 3.1 组织可以限制登录方式

`forced_login_method` 可要求 ChatGPT 或 API 登录；`forced_chatgpt_workspace_id` 可限制允许的 workspace。登录成功不代表一定符合当前配置，runtime 还会执行 `enforce_login_restrictions`。

### 3.2 凭据存储

`cli_auth_credentials_store` 支持：

- `file`
- `keyring`
- `auto`
- `ephemeral`

`auto` 优先 OS keyring，失败时可回退文件。文件模式使用 `$CODEX_HOME/auth.json`。选择 keyring 后应确认操作系统服务可用；某些容器或无桌面 Linux 环境最终仍会回退文件。

登录状态和退出：

```bash
codex login status
codex logout
```

## 4. 交互式 TUI

最小启动：

```bash
cd /path/to/repository
codex
```

可直接附带初始任务：

```bash
codex "解释这个项目的入口和测试方式"
```

启动前应确认：

- cwd 是预期仓库。
- 项目配置和 AGENTS 是否可信。
- 当前 permission profile/sandbox。
- 是否连接 remote app-server。

TUI 当前通过 app-server session 管理 thread。它可使用 embedded、local daemon 或 remote target，因此“终端在本机”不总意味着工具在本机执行。

## 5. Permission、approval 与 sandbox

初学者应先理解三个不同问题：

| 层 | 问题 |
| --- | --- |
| Permission profile | 这个会话声明允许哪些文件、进程、网络能力？ |
| Approval | 某个动作是否需要用户确认？ |
| Sandbox | OS/runtime 实际如何限制副作用？ |

常见模式：

- `read-only`：适合先理解代码。
- `workspace-write`：允许在工作区修改，仍限制更广泛副作用。
- `danger-full-access`：风险高，只在明确受控环境使用。

`--dangerously-bypass-approvals-and-sandbox` 会同时绕过两类保护，不能当作普通排障选项。

旧 `--full-auto` 只是隐藏的兼容陷阱，会提示改用 `--sandbox workspace-write`。

## 6. 非交互 `codex exec`

最小用法：

```bash
codex exec "运行测试并解释失败原因"
```

通过 stdin：

```bash
printf '%s\n' "总结当前 diff" | codex exec -
```

若同时提供 prompt 和 piped stdin，stdin 会作为额外 `<stdin>` 内容追加，而不是悄悄覆盖 prompt。

### 6.1 Git 仓库检查

exec 默认拒绝在非 Git/不可信目录运行，除非：

```bash
codex exec --skip-git-repo-check "..."
```

这不是 sandbox 替代品。它只表示用户明确接受非仓库工作目录。

### 6.2 结构化输出

```bash
codex exec --json "..."
codex exec --output-schema schema.json "..."
codex exec -o final.txt "..."
```

- `--json` 输出逐行事件。
- `--output-schema` 约束最终响应形态。
- `--output-last-message` 另存最后一条 agent 消息。

机器应解析 JSONL event type，不应对人类文本做正则。

### 6.3 Headless approval

exec 首先构造 `AskForApproval::Never`，因为无人值守任务不能停在交互提示上。但如果最终配置启用 auto-review reviewer，构建逻辑会重新解析 approval policy，使自动审查接管。

这说明 headless 安全不是固定一个 enum，而是 permission、sandbox、reviewer 和 repo check 的组合。

### 6.4 无法处理的交互

exec 对某些 server→client 请求必须采取非交互策略。例如 MCP elicitation 默认自动取消。需要复杂 OAuth/表单交互的任务更适合先在交互客户端完成配置。

## 7. 第一次任务的推荐流程

### 只读理解

```text
1. 在仓库根目录启动。
2. 选择 read-only。
3. 让 Codex读取 AGENTS、README 和构建文件。
4. 要求给出计划，不修改文件。
5. 核对引用路径。
```

### 小范围修改

```text
1. 确认 Git 工作区状态。
2. 使用 workspace-write。
3. 明确允许修改的文件。
4. 要求运行针对性测试。
5. 检查 diff 后再提交。
```

### CI/脚本

```text
1. 固定 Codex 版本。
2. 显式设置 cwd、model、sandbox 和 config。
3. 使用 --json。
4. 设置进程 timeout。
5. 保存 thread id 和最终消息。
6. 对非零退出码和 interrupted/failed turn 分开处理。
```

## 8. Resume 与 ephemeral

正常 session 会写 rollout/state，可稍后：

```bash
codex resume
codex resume --last
codex exec resume --last "继续完成剩余任务"
```

不希望持久化时：

```bash
codex exec --ephemeral "..."
```

ephemeral 适合敏感或一次性任务，但代价是无法依赖常规 session 文件恢复。

## 9. 常见故障

### 登录成功但启动仍被拒绝

检查 `forced_login_method`、workspace restriction、cloud managed requirements 和 auth store 中是否混有旧凭据。

### 配置看似未生效

检查 layer source、project trust、denylist、requirements fallback 和 startup warning。

### exec 一直失败

依次检查：

- 是否在 Git 仓库。
- cwd 是否存在。
- permission/sandbox 是否允许。
- output schema 是否有效。
- MCP/工具是否要求交互。
- cloud config 是否 fail closed。

### TUI 找不到旧会话

检查 cwd filter、`--all`、interactive/non-interactive source、state DB 与 rollout 文件。

## 10. 验证命令

```bash
# 登录入口与限制
rg -n "with_api_key|with_access_token|device_code|forced_login_method|forced_chatgpt_workspace_id" \
  codex-rs/cli/src codex-rs/login/src codex-rs/core/src/config

# exec 的输入、JSON、repo check 和 headless policy
sed -n '1,240p' codex-rs/exec/src/cli.rs
rg -n "skip_git_repo_check|AskForApproval::Never|McpServerElicitation" \
  codex-rs/exec/src/lib.rs

# TUI app-server target
rg -n "AppServerTarget|RemoteAppServerClient|start_embedded_app_server" \
  codex-rs/tui/src/lib.rs

# 登录集成测试
just test -p codex-cli login
```

## 小结

初级使用的目标不是尽快打开“全自动”，而是建立可解释的运行链：

```text
正确 binary
  -> 合规 auth
  -> 正确 cwd/trust
  -> 明确 permission/approval/sandbox
  -> 可解析输出
  -> 可恢复 thread
```

这条链稳定后，Codex 才适合进入更复杂的扩展、远程环境和云任务工作流。
